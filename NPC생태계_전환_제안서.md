# NPC 생태계 전환 제안서

**작성일**: 2026-07-02
**대상 프로젝트**: simulation (마을 성장 시뮬레이터)
**목표**: "작업을 실행하는 도구"처럼 보이는 NPC를, 스스로 살아가며 마을 생태계를 키워나가는 존재로 전환한다.

---

## 1. 현상 진단 — 왜 "단순 반복 작업 도구"처럼 보이는가

현재 코드를 기준으로 원인을 짚으면 다음 다섯 가지로 요약된다.

### 1-1. 중앙 계획 구조 (가장 큰 원인)
`TaskBoardSystem`이 매 틱 마을 전체의 작업 목록(farm_work, gather_wood, build …)을
중앙에서 생성하고, `AgentDecisionSystem`은 그 목록에서 효용 점수가 가장 높은 작업을
집어간다. 구조적으로 NPC는 **"마을 정부가 발주한 일감을 처리하는 일꾼"**이다.
아무리 점수 계산에 성격(traits)이 섞여 있어도, 일감 자체가 위에서 내려오는 한
"자율적으로 살아간다"는 인상은 생기지 않는다.

### 1-2. 개인의 삶이 없다
- **소유가 없다**: 모든 자원이 `state.resources`(마을 공용 풀)에 들어간다.
  `Citizen.wealth`와 `money` 자원 타입이 이미 정의돼 있지만 사용되지 않는다.
  내 것이 없으니 "내 살림을 꾸린다"는 서사가 불가능하다.
- **기억이 없다**: `decisionReasons`는 이번 결정의 점수표일 뿐, NPC에게
  과거("작년 겨울에 굶었다", "친구가 병으로 죽었다")가 남지 않는다.
- **장기 목표가 없다**: goal은 전부 단기 행동(eat, rest, work_farm)이고,
  "대장장이가 되고 싶다", "집을 갖고 싶다" 같은 인생 목표가 없다.

### 1-3. 관계가 없다
`Citizen`에 traits/skills/specialty는 있으나 **다른 NPC를 가리키는 필드가 하나도 없다**
(가족 링크, 친구, 원한 전무 — `groupId`만 존재). 출산 시스템이 있는데도 부모-자식
관계가 데이터로 남지 않는다. 생태계의 핵심은 개체 간 상호작용인데, 지금 NPC들은
서로에게 투명인간이다. `care_sick`조차 "환자"라는 익명의 작업 대상일 뿐이다.

### 1-4. 모두가 같은 공식으로 산다
전 주민이 `chooseGoal()`의 동일한 점수 공식을 돌린다. 성격은 ±0.1~0.3 수준의
가중치 미세조정이라 행동이 수렴한다. 결과적으로 "농부 5명이 동시에 농장으로 출근 →
동시에 퇴근"하는 공장 교대 패턴이 나온다.

### 1-5. 환경이 반응하지 않는다
나무·돌은 목표 재고량(`woodStockTarget`)까지 채우는 무한 수도꼭지다. 남획하면
고갈되고, 놔두면 회복되는 **자연 쪽 피드백 루프가 없어서** "생태계"의 절반이 비어 있다.

> **한 줄 요약**: 지금은 「중앙 계획 + 익명 노동자 + 무한 자원」 구조다.
> 생태계처럼 보이려면 「환경의 기회 + 기억·관계를 가진 개인 + 유한한 자연」으로
> 뒤집어야 한다.

## 2. 전환 원칙

1. **질서는 아래에서 위로**: 마을 단위 질서(직업 분화, 시설 건설)는 개인 결정의
   합으로 창발해야 한다. 중앙 시스템은 "지시"가 아니라 "신호"만 낸다.
2. **모든 행동에는 개인적 이유가 있다**: NPC가 왜 그 일을 하는지, 자기 상태·기억·
   관계로 설명 가능해야 하고, 그 이유가 UI에 보여야 한다.
3. **창발은 보여야 존재한다**: 내부적으로 아무리 복잡해도 관찰 UI가 없으면
   플레이어에게는 존재하지 않는 것과 같다. 시뮬레이션 개선과 관찰 UI를 항상 짝으로 만든다.
4. **기존 자산 재사용**: traits/skills/specialty/wealth/groupId/SeededRandom 결정성 등
   이미 있는 뼈대를 최대한 살린다. 전면 재작성이 아니라 의미 재배치가 목표다.

## 3. 단계별 개선안

### Phase 1 — 개인의 탄생: 기억·목표·소유 (기반 작업)

| 항목 | 내용 | 관련 기존 코드 |
|---|---|---|
| 에피소드 기억 | NPC별 `memories: LifeEvent[]` 추가. 굶주림·질병·출산·친구의 죽음·전직 등을 날짜와 함께 기록. 최근 기억이 결정 가중치에 반영(굶어본 자는 식량 비축 성향↑) | `Citizen` 타입, `AgentDecisionSystem` |
| 인생 목표(aspiration) | 성년 시 성격 기반으로 1개 부여: 기술 장인, 가정, 부, 탐험 등. 장기 목표가 일상 선택에 지속 보정치를 준다 | `PopulationFactory`, traits |
| 개인/가구 소유 | 생산물 일부를 개인·가구 인벤토리로. 집 = 가구 저장소(겨울 모드의 `house.winter.firewoodStored`가 이미 선례) | `wealth`, `ResourceInventory` |
| 일대기 패널 | NPC 클릭 → 이름·나이·가족·기억 타임라인·현재 목표·"지금 이 일을 하는 이유"를 보여주는 UI | `decisionReasons` 재활용 |

**기대 효과**: 관찰자가 특정 NPC를 "따라가고 싶어"진다. 같은 작업을 해도
"흉년의 기억 때문에 식량을 모으는 A"와 "장인이 꿈이라 대장간을 기웃대는 B"로 읽힌다.

### Phase 2 — 관계망: 사회 생태계

| 항목 | 내용 |
|---|---|
| 가족 그래프 | 출산 시 부모-자식 링크 저장, 배우자 관계 도입. 가구(household) 단위 생활(같은 집, 식량 공유) |
| 관계 점수 | `relationships: Map<citizenId, {affinity, type}>`. 같은 작업장 근무, 간병(care_sick), 음식 나눔에서 자연 축적. 성격 충돌 시 반감도 축적 |
| 사교 행동 | 유휴 시간(현 `wander`)을 사교로 승격: 잡담, 함께 식사, 선물. 행복·관계에 영향 |
| 관계가 결정에 개입 | 친구 따라 직업 선택, 가족이 아프면 만사 제치고 간병, 사이 나쁜 사람과는 협업 회피 |
| 관계망 시각화 | 마을 화면 오버레이 or 패널로 가문 트리·친밀도 네트워크 표시 |

**기대 효과**: "왜 저 둘은 항상 같이 다니지?", "저 가문이 마을 대장간을 독점하네"
같은 이야기가 생긴다. 생태계 느낌의 절반은 개체 간 상호작용에서 나온다.

### Phase 3 — 분권화: 중앙 작업판 → 환경의 기회 신호

핵심 구조 전환. `TaskBoardSystem`을 폐기하지 않고 **의미를 뒤집는다**.

- **작업(지시) → 기회(광고)**: 건물·자원지가 "여기 일하면 이런 보상이 있다"는
  기회(opportunity)를 게시한다. 보상은 개인 소유물(식량, 임금)로 지급.
- **가격 신호로 분업 창발**: 자원 희소성이 보상 크기를 결정. 식량이 귀해지면
  농사 보상이 올라 사람이 몰리고, 남으면 떠난다. 지금은 설계자가 튜닝한
  priority 숫자가 하던 일을 시장이 대신한다. (`ProfessionEmergenceSystem`의
  수요 기반 기회 점수 개념을 개인 차원으로 내리는 것)
- **전직 = 개인의 결심**: "대장장이 수요 발생 → 시스템이 배치"가 아니라,
  "벌이가 시원찮던 C가 대장간 기회를 보고 전직을 결심"으로. 결심은 기억에 기록.
- **NPC 간 거래**: 시장 건물에서 개인 잉여물 교환. `money` 자원 활성화.

**기대 효과**: 직업 분포가 환경 조건에 따라 시드마다 다르게 흘러간다.
"이 마을은 사냥꾼이 많아졌네"가 설계가 아니라 역사적 결과가 된다.

### Phase 4 — 살아있는 환경: 자연 생태계

- **재생 자원**: 나무는 베면 줄고 시간이 지나면 자란다(성장 단계 렌더링).
  구역별 고갈 → 벌목공이 더 먼 숲으로 이동하거나 전직 → 마을 형태가 바뀐다.
- **토양 비옥도**: 연작 시 수확 감소, 휴경 시 회복. 농장 이전·개간의 동기.
- **사냥감 개체수**: 수렵 압력에 반응하는 간단한 개체군 모델(로지스틱 성장).
- **계절 순환**: 겨울 시나리오의 자산(체온·땔감·난방)을 일반 성장 모드의
  반복 계절로 통합. 매년 겨울이 "생존 이벤트"로 돌아오면 비축·기억·사망이
  자연스럽게 세대 서사를 만든다.

**기대 효과**: 인간 활동 ↔ 자연 반응의 양방향 피드백이 생겨 말 그대로의
"생태계"가 완성된다. 남획→고갈→위기→적응의 드라마가 개입 없이 발생한다.

### Phase 5 — 관찰 UX: 창발을 이야기로 보여주기 (전 단계와 병행)

- **마을 연대기(Chronicle)**: "D12: 첫 대장장이 탄생(C, 벌목 흉년이 계기)",
  "D23: 가혹한 겨울로 3명 사망" 등 굵직한 사건을 자동 기록하는 이벤트 피드.
  이벤트 버스는 로드맵에 이미 신규 항목으로 잡혀 있음.
- **말풍선 고도화**: 현 `CitizenThoughtSystem`의 상태 라벨("배고파")을
  맥락 서사로 확장("올겨울 땔감이 걱정이야", "아버지처럼 농부가 될 거야").
- **통계·타임랩스**: 직업 분포 변화 그래프, 자원-인구 생태 곡선, 가문 트리,
  세대 교체 타임라인. 배속/일시정지로 "관찰하는 재미"를 지원.
- **따라가기 카메라**: NPC 1명을 선택해 하루를 따라다니는 관찰 모드.

## 4. 추진 순서 권고

효과 대비 구현 비용 기준으로 다음 순서를 권한다.

1. **Phase 1 (기억+일대기 패널)** — 구조 변경 없이 `Citizen` 확장과 UI만으로
   체감이 크게 바뀐다. 최소 비용으로 방향 검증 가능. ★착수 추천
2. **Phase 2 (가족·관계망)** — 출산 시스템이 이미 있어 가족 링크부터 시작하면 저렴.
3. **Phase 5 (연대기·말풍선)** — 1·2에서 쌓이는 데이터를 보여주는 창구. 병행 개발.
4. **Phase 3 (분권 경제)** — 가장 큰 구조 전환. 1·2·5로 개인이 서사를 가진 뒤에
   해야 효과가 보인다. 겨울 모드 밸런스(평균 생존 ~12명)에 영향이 크므로
   성장 모드에서 먼저 적용 후 겨울 모드에 이식.
5. **Phase 4 (자연 재생)** — 독립성이 높아 언제든 끼워넣을 수 있으나,
   분권 경제(가격 신호)가 있어야 고갈→적응 반응이 극적으로 보인다.

## 5. 성공 판정 기준

전환이 성공했는지는 다음 질문으로 판정한다.

- [ ] 서로 다른 두 시드가 **뚜렷이 다른 역사**(직업 구성, 가문, 위기와 극복)를 만드는가?
- [ ] 플레이어가 **특정 NPC 개인을 따라가고 싶어**지는가? (일대기가 읽을 만한가)
- [ ] NPC의 모든 행동에 대해 "왜?"를 물으면 **개인적 이유**(기억·관계·목표)가 나오는가?
- [ ] 30일 무개입 관찰 시 직업 분포·마을 구조가 **환경 조건에 따라** 달라지는가?
- [ ] 개발자가 밸런스 숫자를 고치지 않아도 자원 위기가 **스스로 조정**되는가?

## 6. 리스크와 대비

| 리스크 | 내용 | 대비 |
|---|---|---|
| 밸런스 붕괴 | 공용 풀 → 개인 소유 전환 시 분배 실패로 아사 연쇄 가능(과거 unstick 버그처럼 붕괴 원인이 한 곳에서 시작) | 가구 단위 저장을 중간 단계로. 마을 창고를 "최후 안전망"으로 유지하며 점진 전환 |
| 성능 | 관계 그래프는 O(n²). 인구 50명+에서 틱 비용 증가 | 관계 수 상한(NPC당 8~12), 갱신 주기 분산(틱마다 일부만) |
| 결정성 훼손 | 다중 시드 검증 체계가 결정성에 의존 | 신규 랜덤은 전부 기존 `SeededRandom` 경유, Map 순회는 정렬 후 |
| 겨울 모드 회귀 | 결정 로직 변경이 생존 밸런스를 흔듦 | 성장 모드 우선 적용, 겨울 시나리오는 기존 경로 유지 후 단계 이식. 기존 테스트(`MountainWinterScenario.test.ts`) 통과를 게이트로 |
| 복잡도만 늘고 안 보임 | 내부 시뮬레이션만 정교해지고 관찰자는 차이를 못 느낌 | 모든 시뮬레이션 작업에 대응 UI 작업을 짝으로 배정(원칙 3) |

## 부록: 재사용 가능한 기존 자산 매핑

| 기존 자산 | 새 역할 |
|---|---|
| `Citizen.traits / skills / specialty` | 기억·목표·관계 형성의 성향 입력값 (이미 충분히 풍부) |
| `Citizen.wealth`, `money` 자원 (미사용) | Phase 3 개인 경제의 화폐 |
| `Citizen.groupId` | 가구(household) ID로 의미 확정 |
| `decisionReasons` | 일대기 패널의 "지금 이 일을 하는 이유" 표시 |
| `ProfessionEmergenceSystem` 기회 점수 | 개인 차원 전직 결심의 입력 신호로 격하 |
| `TaskBoardSystem` | OpportunityBoard(기회 게시판)로 의미 전환 — 코드 골격 유지 |
| `CitizenThoughtSystem` | 맥락 서사 말풍선의 토대 |
| 겨울 시나리오(체온·땔감·질병) | Phase 4 계절 순환의 겨울 파트로 통합 |
| `house.winter.firewoodStored` | 가구 단위 저장소의 선례 → 일반화 |
