# 사회 실험형 마을 시뮬레이터 — 1단계 구조 검토 보고서

작성일: 2026-06-25
대상: 1차 구현(요청서 28번 "첫 작업 요청"까지) 결과물
검토 방법: 전체 소스 정독 + 헤드리스 100일/500일/2000일 실행, 다중 시드(6종) 및
생산성·창고·초기조건 스트레스 조건 실측

---

## 종합 결론

엔진/렌더 분리와 결정론(재현성)은 매우 견고하다. 1단계 목표는 충실히 달성되었다.
다만 "사회 실험형"이라는 정체성 관점에서 핵심 결함이 하나 있었다 —
**시드가 무엇이든 항상 똑같은 결과(완전 번영)로 수렴**하고, 생산성 임계값을 살짝 넘으면
**전원 사망**으로 떨어지는 이분법 구조였다. 즉 "다양한 사회적 결과 관찰"이라는
핵심 가치가 아직 나오지 않았다.

이 보고서의 7개 항목 분석 결과를 정리하고, 끝에 P0/P1 조치 내역을 덧붙인다.

---

## 1. 엔진 ↔ Phaser/React 분리 — 우수

- `src/simulation/` 전체가 React/Phaser를 한 줄도 import 하지 않는다.
  `index.ts`는 순수 타입/클래스만 export 한다.
- 렌더 계층(VillageScene, VillageCanvas)은 엔진이 만든 읽기 전용 스냅샷만 소비한다.
  `getSnapshot()`이 시민·건물·통계를 깊은 복사해 넘기므로 렌더가 엔진 상태를 오염시킬 수 없다.
- 헤드리스 스크립트와 테스트가 DOM 없이 엔진만 구동 → 분리가 실제로 검증된다.

우려(경미): App.tsx의 requestAnimationFrame 루프가 엔진 시간 진행을 직접 운전한다.
1단계는 괜찮으나, 향후 Web Worker로 계산을 옮기려면 이 RAF→advanceRealTime 결합을
메시지 기반으로 바꿔야 한다.

## 2. 동일 시드 재현성 — 우수

- SeededRandom(FNV-1a 해시 + Mulberry32)이 유일한 난수원이며 Math.random() 사용처가 없다.
- 헤드리스 실측: reproducible=true (100일 스냅샷 JSON 완전 일치).
- 잠재 리스크: 정렬 의존 결정론. WorkforceSystem은 동률을 id.localeCompare로 깨서 안정적이다.
  앞으로 컬렉션 순회를 정렬 없이 추가하면 재현성이 미묘하게 깨질 수 있으므로
  "모든 컬렉션 순회는 결정적 순서 보장"을 팀 규칙으로 둘 것.

## 3. 식량 생산·소비 장기 순환 — 순환이 아니라 "포화 후 정체"

500/2000일 실측 결과, 식량은 순환하지 않고 창고 상한에 붙어 고정되었다.
(100일 이후 재고 500 고정, 농부 20·농장 2, 평균 배고픔 0)

구조적 원인 2가지:
1. 농장이 생산에 영향을 주지 않았다. FoodSystem의 생산량은 `농부 수 × 생산성`일 뿐
   농장 개수·정원과 무관했고, 농장·workers·capacity는 사실상 장식이었다.
2. 창고 상한이 "소비 전"에 적용된다. min(상한, 재고+생산) 후 소비하므로
   하루 끝 실효 비축 = 창고용량 − 하루소비량이며, 충격이 없어 비축은 평생 쓰이지 않았다.

장기 순환(계절·흉작·가격 변동)은 1단계 범위 밖이라 버그는 아니나,
"장기 정상 순환"의 답은 "순환은 아직 없고 단조 포화"였다.

## 4. 무조건 붕괴/번영 회피 — 미흡 (최우선 개선 대상이었음)

여러 시드(village-001, a, b, c, experiment-x, zzz)를 500일 돌린 결과가 소수점까지 동일했다:
  모든 시드 → pop=100  food=500  farmers=20  happy=80.2  hunger=0  이탈=0

생산성 파라미터를 조금만 바꾸면 절벽처럼 갈렸다:
  foodPerFarmerPerDay 5.3~1.05 → 전원 생존·완전 번영(동일 상태)
  foodPerFarmerPerDay 1.0 이하  → 전원 사망(pop=0)

원인: 목표 농부 수 = ceil(인구 × 1.05 / 생산성). 생산성이 1.05 미만이면 전원이 농부여도
잉여 버퍼를 못 만들고, 인구는 출생·이민이 없어 단조 감소만 가능 → 한 번 무너지면 회복 불가.
결국 "강한 단일 안정점(완전 번영) + 임계값 아래는 전멸"이라는 이분 구조였다.

## 5. 100배속에서 결과가 달라질 가능성 — 없음 (안전)

- SimulationClock는 실시간 Δ에 배속을 곱해 누적하고 정수 일수만 실행, 잔여는 이월한다.
  stepDay()는 며칠을 한꺼번에 처리하든 하루씩 도는 것과 수학적으로 동일하다.
- 따라서 1배속·100배속·헤드리스가 같은 일자 시퀀스 → 같은 결과. 차이 없음.
- App.tsx의 Math.min(250, elapsed) 클램프는 백그라운드에서 체감 속도만 늦출 뿐
  일자 계산의 결정성은 유지된다.

## 6. 경제·건설·이민 확장 적합성 — 부분적, 선제 리팩터 권장

- (a) stepDay가 하드코딩 파이프라인. 세금·시장·범죄·이민마다 이 메서드를 직접 수정해야 했다.
- (b) 자원·직업·건물이 단일 값 고정. food만/ farmer|unemployed만 → 나무·돌·돈 추가 시 다파일 수정.
- (c) 밸런스 상수가 코드에 박힘(FoodSystem의 55, 0.08, 0.25, 34, 0.48, 7, -4, 28 등).
- (d) 통계 누적 + 전량 깊은복사. getSnapshot이 매 프레임 전체 통계를 깊은 복사 → 장기 검증에서 악화.
- (e) 이벤트/타임라인 인프라 부재.
- (f) traits·groupId·wealth·age가 생성만 되고 미사용(확장 훅은 마련됨).

## 7. 수정 필요 파일과 우선순위

P0  FoodSystem.ts + WorkforceSystem.ts — 절벽형 이분 결과 완화(초기조건 분산·확률 요동·
    농장이 실제 생산을 좌우), 행복/건강 상수 외부화
P0  SimulationConfig.ts — 행복/건강 공식 상수 외부화
P1  SimulationEngine.ts — stepDay 시스템 파이프라인화, getSnapshot 전체 통계 깊은복사 제거
P1  types.ts — 자원 맵 일반화, 직업/건물 enum 확장 여지
P2  (신규) 이벤트/타임라인 모듈
P2  SimulationEngine.test.ts — 다중 시드 분기(divergence) 및 1,000일 안정성 테스트 추가

## 부록 A. P0/P1 조치 내역 (이번 작업에서 반영)

### P0 — 결과 다양성 + 밸런스 외부화
1. 마을별 토지 비옥도(landFertility) 도입: 시드에서 파생된 비옥도를
   `[landFertilityMin, landFertilityMax]` 범위에서 뽑아 farmer당 생산성에 곱한다.
   → 같은 base 설정이라도 시드별로 농부 수·식량·행복·인구 궤적이 달라진다.
2. 일일 생산 요동(dailyProductionNoise): 매일 ±noise 비율의 확률 변동을 생산에 적용.
3. 농장이 실제 생산을 좌우: 생산 인력을 농장 총정원으로 상한 처리.
   농장이 모자라면 잉여 농부는 생산에 기여하지 못한다.
4. 행복/건강/생산성 공식 상수를 전부 SimulationConfig로 외부화(데이터 기반 튜닝 가능).

### P1 — 확장성 + 장기 성능
5. stepDay를 시스템 파이프라인으로 리팩터링: 각 시스템이 (ctx)=>void 형태.
   경제·건설·이민·범죄 시스템을 배열에 추가하는 것만으로 확장 가능.
6. getSnapshot의 전체 통계 깊은복사 제거: 일일 통계는 생성 시 동결(freeze)하고
   스냅샷은 불변 참조를 공유 + 최근 구간(recentStatistics)을 별도 제공.
7. 자원 타입 일반화: food 단일 값 → ResourceType 맵(food/wood/stone/money) 기반
   ResourcePool/VillageResources/ResourceInventory. 향후 자원 추가가 타입 한 곳 수정으로 끝난다.

### 검증
- 기존 단위/통합 테스트 전부 통과 유지(재현성·식량흐름·농장수요·주택수요·기아·헤드리스).
- 신규: 서로 다른 시드가 서로 다른 궤적을 만든다는 분기 테스트, 1,000일 안정성 테스트 추가.
- 헤드리스 다중 시드 재실행으로 "시드별 결과가 실제로 달라짐"을 확인.
